# 2. CLI + harness over the HTTP API; no web dashboard

Date: 2026-06-17

## Status

Accepted

## Context

The assignment requires a way to register endpoints, ingest events, a test
harness, and observability. It explicitly allows the interface to be "an HTTP API,
a CLI harness, or some other interesting idea," and allows observability to be
"logs, metrics, status API, a dashboard, or something totally different."

An initial plan included a Vite + shadcn/ui dashboard with tabs and action
buttons. Within a 2–6 hour budget, a polished frontend is the highest
effort-per-point item and the most likely to be left half-finished — which the
rubric penalizes more than a deliberately narrower scope.

## Decision

Cut the web dashboard. Build:

- **One daemon** (`bun run dev`): Hono serving the ingest API, sink routes, the
  status/metrics API, and the in-process worker.
- **A CLI** that is a *thin client over the HTTP API* — never touching SQLite
  directly. Commands to add endpoints, create sinks, emit events, list
  deliveries/attempts, and show status.
- **A harness script** that drives the same HTTP API to run end-to-end scenarios
  and print a report.
- **Observability** = structured JSON logs from the daemon, a status/metrics JSON
  endpoint, and a `status --watch` CLI view.

The CLI talks to the API, not the database, so there is a single writer (the
daemon owns the DB), and the CLI, harness, and any external consumer all exercise
the same surface.

## Consequences

- Removes the largest time sink; effort concentrates on the delivery engine,
  reliability, and tests — the heart of the assignment.
- The "where do you invest in UX polish?" signal moves to the CLI and status
  output, which must therefore be genuinely polished (e.g. a live `--watch` view),
  not raw dumps.
- No direct-DB CLI path avoids two writers to one SQLite file and split-brain.
- **Future work:** the status/metrics API is UI-ready, so a dashboard could be
  added later without changing the core.

Question 12 — Observability: what's actually in status/metrics, and what do the structured logs capture?

Now that the CLI status view is your observability story (no dashboard to lean on), let's define what it shows. Don't over-build here — pick the few numbers that tell the system's health at a glance.

(a) The status/metrics endpoint. My recommendation — a small, queryable JSON snapshot computed from the tables: - Queue depth: counts of deliveries by status (pending, processing, delivered, failed). - Throughput: deliveries completed in the last minute/5 min. - Success rate: delivered ÷ (delivered + failed) over a window. - Retry health: count currently in backoff, and a distribution of attempts-to-success. - Latency: p50/p95 of Attempt duration_ms (you're already storing it from Q7). - Endpoints: total, and how many are auto-disabled (from 410).

(b) webhooks status --watch: re-poll that endpoint on an interval and re-render — so you literally watch pending drain and failed tick up during a harness run. This is the demo money-shot.

(c) Structured logs. My rec: JSON lines, one per significant lifecycle event — event.ingested, delivery.created, attempt.started/succeeded/failed, delivery.exhausted, endpoint.disabled — each with the relevant IDs, status code, duration, and attempt number. Correlatable by delivery_id/event_id. This is cheap (a tiny logger) and it's what makes failures debuggable, which the assignment explicitly asks about ("what happens when things fail in weird ways").

The taste decision: compute metrics on-read from the tables, don't maintain counters. At this scale a few GROUP BY queries are instant, and avoiding a parallel counter system means no drift and less code.

Confirm: that metrics set (queue depth / throughput / success rate / retry health / latency / endpoints), computed on-read, plus JSON-lines lifecycle logs, plus --watch? Or do you want to trim the metrics list?

I'd take (a)+(c) — and here's the reasoning for picking logs over the live view:

  • (c) is a second, independent observability pillar; (b) is polish on top of (a). The status endpoint already gives you the glanceable health story — --watch just re-renders it on a timer. Useful, but it doesn't tell you anything status run twice doesn't. Logs tell you something the metrics can't: the per-Delivery story of why something failed.
  • The assignment lists "logs" first and explicitly asks "what happens when things fail in weird ways" — structured lifecycle logs are the direct answer to that prompt. Aggregate metrics show that failures happened; logs show what happened to a specific Delivery.
  • Logs pay for themselves while you build. You'll be debugging the worker, the retry classifier, the HMAC — having attempt.failed {delivery_id, status_code, error} lines from hour one makes that work faster. --watch only helps at demo time.

So: (a)+(c) — metrics API + JSON-lines lifecycle logs — and keep --watch as a clearly-labeled stretch goal if you finish the spine early. That way your one-shot webhooks status still gives the visible health snapshot; you just don't commit time to the animated version.

Locked. We've covered the spine end to end. Let me close with the edge cases — the assignment explicitly rewards "what happens when things fail in weird ways," and these are cheap to decide now and awkward to retrofit.

Question 13 — Endpoint lifecycle vs. in-flight Deliveries. Three concrete scenarios:

Scenario 1 — Endpoint deleted while it has pending/processing Deliveries. An operator removes an Endpoint, but 40 Deliveries to it are sitting pending or mid-retry. What happens to them? My rec: soft-delete the Endpoint (deleted_at), and the worker skips/cancels Deliveries whose Endpoint is deleted (mark them canceled, a distinct terminal state — not failed, since nothing failed). Soft-delete also keeps your historical Deliveries/Attempts referentially intact for the audit trail. Hard-delete would orphan rows and corrupt your metrics history.

Scenario 2 — Endpoint auto-disabled by 410 (from Q5) while Deliveries are queued. My rec: disabled is not deleted — existing Deliveries to it stop (mark canceled or hold), and no new Deliveries are created for it during fan-out. Distinguish disabled (system turned it off, reversible) from deleted (operator removed it).

Scenario 3 — Endpoint's event_types changes after Events were already queued for it. My rec: routing is decided at fan-out time, not delivery time. Once a Delivery exists, it delivers regardless of later subscription changes. Re-evaluating routing on every attempt would be surprising and racy. (This means: fan-out reads the subscription once, when the Event is ingested.)

The through-line: Endpoints are soft-deleted, with disabled and deleted as distinct states, and routing is frozen at fan-out time. This adds one canceled terminal Delivery state and deleted_at/disabled_at columns on Endpoint.

Confirm all three? Or do you want, e.g., hard-delete instead of soft-delete, or routing re-evaluated at delivery time?

### Delivery
One Event headed to one Endpoint. The unit we track and retry. Has a status and a
retry count. One Event ingested with N matching Endpoints produces N Deliveries.

Delivery status values:
- **pending** — waiting to be attempted (or waiting out backoff).
- **processing** — claimed by the worker, attempt in flight.
- **delivered** — terminal; an Attempt succeeded (2xx).
- **failed** — terminal; retries exhausted or a permanent error (e.g. 4xx).
- **canceled** — terminal; the target Endpoint was deleted or disabled before
  delivery. Nothing *failed* — distinct from `failed`.

### Endpoint states
- **active** — receives fan-out and is delivered to.
- **disabled** — turned off by the system (e.g. after a 410 Gone). Reversible. No
  new Deliveries are created; queued ones are canceled.
- **deleted** — soft-deleted by an operator (`deleted_at` set). Rows are retained
  for the audit trail; queued Deliveries are canceled.

Routing is frozen at fan-out time: a Delivery, once created, is unaffected by later
changes to the Endpoint's subscribed Event Types.

Captured. Now your demo question — how do you show off these edge cases? — is a great instinct, because a failure-handling feature nobody can see is worth almost nothing to a reviewer. This is exactly what the harness scenario runner is for. Make the harness a menu of named, self-contained scenarios, each of which sets up, triggers, and then prints the resulting state so the behavior is legible without a UI.

My recommendation — the harness exposes named scenarios (bun run harness <name> or a menu), each ending with a printed before/after of the relevant Deliveries + a status snapshot:

  • happy-path — 3 endpoints across types, fire matching events, watch all reach delivered.
  • retry-recovery — a fail-then-recover sink (500×2 then 200); print the Attempt log showing failed, failed, delivered with the backoff timestamps. This is the headline demo.
  • permanent-failure — an always-400 sink; show the Delivery going straight to failed with no wasted retries.
  • gone-disables — a 410 sink; show the Endpoint flip to disabled and queued Deliveries flip to canceled.
  • delete-cancels — fire events, then delete an Endpoint mid-flight; show its pending Deliveries become canceled while others continue.
  • signature-verified — a verify-signature sink that returns 200 only on a valid HMAC; show it succeed, then a tampered-secret variant rejected.
  • routing — endpoints with disjoint event_types; fire one event; show only matching endpoints get Deliveries.

The taste move: each scenario prints the Attempt timeline and the relevant status counts at the end, so the terminal output is the proof. That, plus your bun test suite asserting the same outcomes, means you've both demonstrated and verified every failure mode — directly hitting the "things fail in weird ways" rubric line. The scenarios and the integration tests can even share the same setup helpers.
