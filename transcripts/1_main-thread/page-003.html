# 1. SQLite as the delivery queue with an in-process polling worker

Date: 2026-06-17

## Status

Accepted

## Context

The service must deliver Events to Endpoints reliably (at-least-once), survive
restarts, and support retries with backoff. We need a mechanism that moves a
Delivery from `pending` to a terminal state and re-attempts on failure.

Options considered:

- **External broker** (Redis/BullMQ, RabbitMQ, SQS). Battle-tested queue
  semantics, but adds infrastructure a reviewer must stand up, and works against
  the goal of a single, locally-debuggable binary.
- **In-memory queue.** Trivial to write, but loses all in-flight Deliveries on
  restart — directly undercuts the persistence and delivery-guarantee goals.
- **SQLite as the queue.** The `deliveries` table *is* the queue. An in-process
  worker polls for due rows, claims them, delivers, and writes results back.

We are already using SQLite for persistence, the system is single-process, and
backoff maps naturally onto a `next_attempt_at` timestamp column.

## Decision

Use the `deliveries` table as the queue, driven by an **in-process polling
worker** started alongside the Hono server (one entrypoint).

Each tick the worker atomically claims a batch (`pending` rows that are due, plus
`processing` rows past a visibility timeout) by setting `status='processing'` and
`claimed_at=now`, delivers them concurrently (with a cap), and writes back the
outcome. Backoff is a future `next_attempt_at`. A **visibility timeout** reclaims
rows stranded in `processing` by a crash, preserving the at-least-once guarantee.

## Consequences

- No external infrastructure; `bun run dev` starts API + worker together.
- The queue is just a table — observability and the dashboard are plain `SELECT`s.
- Throughput is bounded by one process and SQLite write throughput. Acceptable for
  this assignment.
- **Future work:** the worker is a separate module that could run as its own
  entrypoint against the same DB for horizontal scaling; a real broker could be
  swapped in behind the same Delivery interface.

Question 6 — Authentication: there are two sides, and they're different problems.

The assignment lists "authentication" as a production concern. For a webhook service that splits into two distinct questions:

(a) Outbound — proving to the Endpoint that the request really came from you. The standard is HMAC payload signing: each Endpoint has a secret (generated at registration), and every Attempt includes a signature header (e.g. X-Signature: sha256=...) computed over the raw request body, plus a timestamp header to prevent replay. The consumer recomputes the HMAC with its copy of the secret and compares. This is what Stripe/GitHub/Svix do, and it pairs naturally with the secret field we already put on Endpoint.

(b) Inbound — who's allowed to call your API (register an Endpoint, ingest an Event)? Options: nothing (open API), a static API key/bearer token, or full auth. For a take-home, full auth (users, sessions) is scope creep that distracts from the delivery engine.

My recommendation: - Outbound: do it. HMAC-SHA256 over the raw body + a timestamp header, signature in X-Webhook-Signature. This is the high-value, on-topic authentication for a delivery service, it's ~20 lines, and your test harness's receivers can actually verify it (great demo: show a receiver rejecting a bad signature). Generate and return the secret once at registration. - Inbound: keep it light. A single static API key checked via middleware (configurable, off by default in dev), and explicitly call it out as "scoped down — real auth/multi-tenancy is future work." Don't build user accounts.

The tension: time budget. HMAC signing is clearly worth it (on-topic, cheap, demoable). Inbound auth is mostly box-checking — I'd do the minimum and say so.

Want HMAC outbound signing + a minimal static-key (or fully open) inbound API? And for inbound: static key, or just leave it open and document it?

Question 7 — Do you store each Attempt as its own row, or just keep a counter + last-response on the Delivery?

We separated Delivery (the durable unit) from Attempt (one HTTP try) in the glossary — but your brain dump's schema only has event_logs with a single response field, which collapses them. Now's the time to decide whether Attempt is a real table or just fields.

Option A — Attempt is just fields on Delivery: attempt_count, last_status_code, last_response, last_attempted_at. Each retry overwrites the previous result. - Pro: one fewer table, simpler queries. - Con: you lose history. The dashboard can't show "this Delivery failed with 503, then 503, then succeeded" — which is exactly the retry story you're building the whole system to tell. You'd be deleting the most interesting observability data.

Option B — Attempt is its own table: attempts(id, delivery_id, attempt_number, status_code, response_body, error, duration_ms, created_at). The Delivery keeps its status/next_attempt_at/attempt_count; each try appends an Attempt row. - Pro: full audit trail per Delivery, a great drill-down in the dashboard, real latency metrics, and it makes the retry behavior visible (which is the point). - Con: one more table and a join.

My recommendation: Option B. The Delivery/Attempt split we made in the glossary should be real in the schema, not just vocabulary. The whole value proposition of this service is reliable retried delivery — storing each Attempt is what lets you show that working, and it's where your metrics (success rate, latency, attempts-to-success) come from. It's one small table.

So the schema becomes: endpoints, events, deliveries (the queue + current status), attempts (the per-try log). Want to go with B?

Question 8 — The Event payload shape: what does a caller send to ingest, and what do you POST to the Endpoint?

Your brain dump flagged this as open ("Validation — need to decide on shape"). Two related shapes:

(a) The ingest shape (what a caller POSTs to your /events endpoint). My rec: a thin envelope —

{ "type": "order.created", "data": { ...arbitrary caller JSON... } }

You validate type against the fixed enum (Q2) and that data is a JSON object. You stay agnostic about what's inside data — that's the caller's domain, not yours. Optionally accept a caller-supplied idempotency_key so re-ingesting the same event doesn't double-fan-out (nice production touch; could be future work).

(b) The delivered shape (what the Endpoint receives). My rec: wrap the event in a delivery envelope so the consumer gets metadata, not just raw data —

{
  "id": "evt_...",          // event id, stable across retries (dedup key from Q3)
  "type": "order.created",
  "created_at": "...",
  "data": { ... }
}

…plus headers: X-Webhook-Id (delivery id), X-Webhook-Timestamp, X-Webhook-Signature (HMAC from Q6). The signature is over this exact serialized body.

The key taste decision: you own the envelope, the caller owns data. That separation is what lets the system be generic while still giving consumers stable, signable, dedup-able metadata.

Two things to confirm: 1. Envelope-with-opaque-data, validated on type + "is an object" — yes? 2. Idempotency key on ingest: include it now, or push to future work?
